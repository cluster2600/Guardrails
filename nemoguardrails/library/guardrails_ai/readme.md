# Guardrails AI Integration for NeMo Guardrails

This directory (`nemoguardrails/library/guardrails_ai/`) provides an integration layer between NeMo Guardrails and the [Guardrails AI](https://github.com/guardrails-ai/guardrails) library. It allows you to leverage validators from Guardrails AI as NeMo Guardrails actions within your conversational AI application.

## Overview

The integration works by wrapping specific Guardrails AI validators (available through `guardrails.hub`) into NeMo Guardrails actions. These actions can then be invoked within Colang flows to validate user inputs or bot outputs. If a validation fails, the flow can be configured to take appropriate action, such as refusing to respond or raising an exception.

## Dependencies

This integration requires the `guardrails-ai` Python package to be installed.

```bash
pip install guardrails-ai
```

**Note:** After installing the `guardrails-ai` package, you also need to install each specific validator you intend to use via the Guardrails AI Hub. This requires an API key, which you can obtain for free by creating an account at [https://hub.guardrailsai.com/keys](https://hub.guardrailsai.com/keys).

Use the `guardrails` CLI to install validators. For example, to install the `toxic_language` validator:

```bash
guardrails hub install hub://guardrails/toxic_language
```

Follow a similar pattern for other validators, replacing `guardrails/toxic_language` with the appropriate validator identifier from the Guardrails AI Hub (e.g., `guardrails/detect_jailbreak`, `guardrails/competitor_check`).

## Available Validators

The following Guardrails AI validators are currently integrated as NeMo Guardrails actions:

1.  **Competitor Check (`validate_guardrails_ai_competitor_check`)**
    *   Checks text for mentions of predefined competitors.
    *   Implemented in: `competitor_check.py`
    *   Corresponding Guardrails AI Validator: [competitor_check)](https://hub.guardrailsai.com/validator/guardrails/competitor_check)

2.  **Detect Jailbreak (`validate_guardrails_ai_detect_jailbreak`)**
    *   Detects potential jailbreak attempts in the input text.
    *   Implemented in: `detect_jailbreak.py`
    *   Corresponding Guardrails AI Validator: [detect_jailbreak](https://hub.guardrailsai.com/validator/guardrails/detect_jailbreak)

3.  **PII Detection (`validate_guardrails_ai_pii`)**
    *   Detects Personally Identifiable Information (PII) in the text.
    *   Implemented in: `guardrails_pii.py`
    *   Corresponding Guardrails AI Validator: [guardrails_pii](https://hub.guardrailsai.com/validator/guardrails/guardrails_pii)

4.  **LLM Provenance (`validate_guardrails_ai_provenance_llm`)**
    *   Checks the provenance or source of information generated by an LLM. *(Further details might be needed based on implementation)*
    *   Implemented in: `provenance_llm.py`
    *   Corresponding Guardrails AI Validator: [provenance_llm](https://hub.guardrailsai.com/validator/guardrails/provenance_llm)

5.  **Restrict to Topic (`validate_guardrails_ai_restrict_to_topic`)**
    *   Ensures the text stays within a predefined set of allowed topics.
    *   Implemented in: `restricttotopic.py`
    *   Corresponding Guardrails AI Validator: [restricttotopic](https://hub.guardrailsai.com/validator/tryolabs/restricttotopic)

6.  **Toxic Language (`validate_guardrails_ai_toxic_language`)**
    *   Detects toxic or harmful language in the text.
    *   Implemented in: `toxic_language.py`
    *   Corresponding Guardrails AI Validator: [toxic_language](https://hub.guardrailsai.com/validator/guardrails/toxic_language)

Each Python file (e.g., `toxic_language.py`) defines a NeMo Guardrails action (using `@action`) that initializes the corresponding Guardrails AI validator and calls its `validate` method. A utility mapping (`guardrails_ai_validation_mapping` from `utils.py`) translates the Guardrails AI result into a format (`{ "valid": True/False }`) understood by the Colang flows.

## Configuration (`config.yml`)

To use this integration, you need to configure it in your NeMo Guardrails `config.yml`:

1.  **Enable the Integration:** Ensure the `guardrails_ai` section exists and `enabled` is set to `true`.

    ```yaml
    # Optional configuration for Guardrails AI
    guardrails_ai:
      enabled: true
      # parameters: # Optional parameters for specific validators
        # detect_jailbreak:
        #   threshold: 0.75
        # toxic_language:
        #   threshold: 0.6
        # guardrails_pii:
        #   redact_fixed_length: 4
        # competitor_check:
        #   competitor_list: ["Competitor A", "Competitor B"]
    ```

2.  **Include Flows:** Add the relevant subflows (`guardrails_ai check input`, `guardrails_ai check output`) to your input/output rails.

    ```yaml
    rails:
      input:
        flows:
          - guardrails_ai check input
          # Optionally add more specific flows from flows.co if defined
      output:
        flows:
          - guardrails_ai check output
          # Optionally add more specific flows from flows.co if defined
    ```

3.  **(Optional) Configure Parameters:** Some Guardrails AI validators accept parameters (like thresholds, lists of competitors, etc.). You can pass these under the `guardrails_ai.parameters` section, keyed by the action name (e.g., `detect_jailbreak`, `toxic_language`). *Note: The Python action implementations must be updated to accept and use these parameters.* The example `config.yml` shows commented-out parameter examples.

## Usage (`flows.co`)

The `flows.co` file defines Colang subflows that orchestrate the validation process:

*   **`run_guardrails_ai_validation(action_name, text)`:** A reusable subflow that takes the action name (e.g., `"validate_guardrails_ai_toxic_language"`) and the text to validate. It executes the action and checks the `valid` field in the result. If invalid, it either raises a `GuardrailsAIException` (if `config.enable_rails_exceptions` is true) or uses the `bot refuse to respond` flow.
*   **`guardrails_ai check input`:** A subflow intended for the input rail. It calls `run_guardrails_ai_validation` for actions relevant to user input (e.g., jailbreak detection, toxic language). You can uncomment or add more validation calls as needed.
*   **`guardrails_ai check output`:** A subflow intended for the output rail. It calls `run_guardrails_ai_validation` for actions relevant to bot output (e.g., PII detection, competitor check). You can uncomment or add more validation calls as needed.

**Example Flow Snippet:**

```colang
# Define the main input check flow
define subflow guardrails_ai check input
  """Check input text using relevant Guardrails AI validators."""
  # Example: Run jailbreak and toxicity checks on input
  execute run_guardrails_ai_validation(action_name="validate_guardrails_ai_detect_jailbreak", text=$user_message)
  execute run_guardrails_ai_validation(action_name="validate_guardrails_ai_toxic_language", text=$user_message)

# Define the main output check flow
define subflow guardrails_ai check output
  """Check output text using relevant Guardrails AI validators."""
  # Example: Run PII and competitor checks on output
  execute run_guardrails_ai_validation(action_name="validate_guardrails_ai_pii", text=$bot_message)
  execute run_guardrails_ai_validation(action_name="validate_guardrails_ai_competitor_check", text=$bot_message)
```

This structure allows you to easily enable/disable specific checks by modifying the `guardrails_ai check input` and `guardrails_ai check output` subflows in `flows.co` and ensuring the corresponding actions are included in the `rails` section of your `config.yml`.
